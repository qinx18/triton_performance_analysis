\begin{abstract}
Emerging programming models like Triton enable developers to better exploit modern accelerators, but translating legacy code to Triton remains challenging. While Large Language Models (LLMs) show promise for code translation, they often generate incorrect or suboptimal implementations due to a lack of precise parallelization reasoning.
%
We present \sysname, a compiler-assisted LLM framework for translating C loop nests to Triton kernels. \sysname extracts parallelization properties using compiler-based analysis and encodes them as structured analysis facts in the LLM prompt, enabling correct and performance-aware parallel code synthesis. A profiling feedback loop further improves performance by feeding GPU hardware utilization metrics back to the LLM for iterative optimization.
%
Across PolyBench/C and TSVC kernels, \sysname achieves near-perfect correctness (100\% and 99\%), improving over LLM-only baselines. On TSVC at realistic data sizes, the generated Triton kernels achieve 1.02$\times$ median performance vs.\ OpenMP GPU offloading compiled by NVIDIA's production compiler on the same GPU. Profiling feedback improves 15 of 30 PolyBench kernels, with gains up to 47$\times$ on individual kernels. We further demonstrate generalization on Rodinia and ECP proxy applications. These results show that combining compiler analysis with hardware profiling feedback enables both correct and high-performance LLM-based parallel code generation.

\section{Introduction}
\label{sec:intro}

Accelerators such as GPUs are now essential for high-performance scientific computing, yet much existing software is still written in legacy C code that is not designed for parallel execution on modern accelerators. Translating these legacy programs to accelerator-friendly implementations is challenging, as it requires identifying parallelism, restructuring computation, and managing synchronization, tasks that require expertise in programming models and hardware.


Emerging programming models such as Triton~\cite{tillet2019triton} simplify GPU programming by providing higher-level abstractions for writing efficient kernels. However, mapping legacy C code to Triton still requires careful reasoning about loop-level parallelism, data dependencies, and execution order. Developers must determine which loop dimensions can be safely parallelized, how memory accesses should be organized, and when synchronization is required - decisions that are subtle and error-prone in practice.


Large Language Models (LLMs) have recently shown promise in code translation, including generating GPU kernels from high-level descriptions~\cite{chen2024chatgpt, ouyang2024kernelbench}. However, LLMs lack precise reasoning about program semantics, especially data dependencies. As a result, they often produce incorrect or inefficient code, including unsafe parallelization that leads to data races, suboptimal kernel structures, and missing synchronization.


In this paper, we present \sysname, a system that augments LLM-based translation with compiler analysis. \sysname extracts parallelization properties from C loop nests using polyhedral analysis (\pet)~\cite{verdoolaege2012polyhedral} and LLVM dependence analysis~\cite{llvm}, and encodes them as structured guidance in the LLM prompt. This allows the model to generate correct and performance-aware Triton code without requiring it to perform dependence analysis itself. 

We evaluate \sysname on a diverse set of benchmarks and application kernels on an NVIDIA RTX~3090 GPU. Across PolyBench/C and TSVC loop kernels, \sysname achieves near-perfect correctness (100\% and 99\%), significantly improving over LLM-only baselines. At realistic data sizes, the generated Triton kernels match the performance of OpenMP GPU offloading compiled by NVIDIA's \texttt{nvc} compiler (1.02$\times$ median on the same GPU). We further demonstrate generalization on kernels from Rodinia and ECP proxy applications. This paper makes the following contributions:

\begin{itemize}
\item A compiler-assisted LLM framework for translating legacy C loop nests to Triton, with a unified analysis module that encodes parallelization constraints as structured facts rather than pattern-specific templates.
\item A profiling-guided optimization loop that feeds GPU hardware utilization metrics (from NVIDIA Nsight Compute) back to the LLM, enabling iterative performance improvement while preserving correctness.
\item Empirical results showing that compiler analysis ensures correctness (100\% on PolyBench/C) and that profiling feedback recovers the performance of hand-tuned prompt templates while maintaining generality.
\end{itemize}

\section{Related Work}
\label{sec:related}

\paragraph{LLM Code Generation for GPUs.}
Chen et al.~\cite{chen2024chatgpt} evaluate ChatGPT on CUDA kernel
generation, finding frequent correctness issues including data races and
incorrect memory management.
KernelBench~\cite{ouyang2024kernelbench} benchmarks LLM-generated
Triton/CUDA on 250 tasks, reporting 20--40\% functional correctness
without guidance.  Our work differs fundamentally: rather than relying
on the LLM's implicit knowledge of parallelism, we provide explicit
compiler-derived constraints, achieving 100\% correctness on
Polybench/C and 99\% on TSVC.

\paragraph{LLM/Agent-Based Kernel Optimization.}
Recent works improve kernel generation through more sophisticated methods.
\emph{RL fine-tuning} approaches train LLMs with execution feedback:
Dr.~Kernel~\cite{liu2026drkernel} uses profiling-based rewards to
avoid reward hacking, achieving 31.6\% of kernels with $\geq$1.2$\times$
speedup on KernelBench (beating Claude~Sonnet~4.5 and GPT-5);
Tehrani et al.~\cite{tehrani2026ftgpt5} RL-fine-tune GPT-5, improving
single-attempt correctness from 43.7\% to 77.0\%;
DICE~\cite{bai2026dice} applies RL to \emph{diffusion} (non-autoregressive)
LLMs, enabling holistic structural planning.
\emph{Search-based} methods keep the LLM frozen:
AVO~\cite{chen2026avo} replaces evolutionary operators with LLM agents,
discovering attention kernels that outperform FlashAttention-4 by 10.5\%;
K-Search~\cite{cao2026ksearch} decouples high-level strategy planning
from code instantiation via a co-evolving world model, achieving up to
14.3$\times$ on MoE kernels;
OptiML~\cite{bhattacharjee2026optiml} uses MCTS with Nsight~Compute
profiler feedback.
KernelBlaster~\cite{dong2026kernelblaster} builds a persistent
cross-task CUDA knowledge base for continual learning without fine-tuning.
For non-GPU targets, AscendKernelGen~\cite{cao2026ascendkernelgen}
and AscendCraft~\cite{wen2026ascendcraft} address Huawei Ascend NPU
kernels through domain-specific datasets and intermediate DSLs.
These works focus on \emph{how} to generate or optimize kernels
(via RL, search, or memory); our work is orthogonal, addressing
\emph{what} parallelism is safe through compiler analysis.
The two directions are complementary: compiler-derived constraints
could serve as reward signals or world-model priors in these frameworks.

\paragraph{Polyhedral Compilation and Auto-Scheduling.}
Polyhedral frameworks such as Pluto~\cite{bondhugula2008pluto} and
PPCG~\cite{verdoolaege2013ppcg} automatically generate parallel code
from affine loop nests.  PPCG targets CUDA directly but is limited to
affine programs and produces low-level code that lacks Triton's
block-level optimizations (automatic memory coalescing, shared memory
tiling).  TVM~\cite{chen2018tvm} and Halide~\cite{ragan2013halide}
use auto-scheduling with learned cost models for tensor operations.
\sysname is complementary: rather than replacing the compiler,
it uses polyhedral analysis to extract \emph{what} parallelism is
safe, then delegates the \emph{how} to the LLM, which can handle
non-affine constructs, irregular access patterns, and high-level
Triton abstractions that polyhedral compilers cannot target directly.

\section{Conclusion}
\label{sec:conclusion}
We have presented \sysname, a system that combines compiler-based analysis with LLM-based kernel generation for accelerators. \sysname encodes parallelization constraints, dependence information, and memory access patterns as structured analysis facts in the LLM prompt, enabling correct and efficient Triton kernel generation from C programs.

\sysname achieves near-perfect correctness (30/30 PolyBench/C, 150/151 TSVC) and generates Triton code that matches production compiler quality: on TSVC at 32\,MB, LLM-generated Triton achieves 1.02$\times$ median vs.\ OpenMP GPU offloading (\texttt{nvc}) on the same GPU. It generalizes to 8 application kernels from Rodinia and ECP proxy apps.
Ablation results show that analysis-guided prompting is essential for correctness, preventing failures and exposing latent race conditions in unguided generation.

Our unified analysis module replaces pattern-specific prompt templates with a general-purpose renderer, reducing prompt logic by 10$\times$ while maintaining correctness. The profiling feedback loop, which feeds NCU hardware metrics back to the LLM, improves 15 of 30 PolyBench kernels and recovers the performance of the pattern-specific approach (1.90$\times$ vs.\ 1.77$\times$ median).
These results suggest that compiler analysis and hardware profiling provide complementary signals---correctness constraints and performance guidance---that could serve as reward functions or world-model priors in RL-based or search-based kernel optimization frameworks.
